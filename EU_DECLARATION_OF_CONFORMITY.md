## Does not conform to

- Export control regulations on cryptographic technology, because this is
  not cryptographic technology. CCL explicitly and loudly states:
  *"CCL provides no cryptographic confidentiality or integrity guarantees
  and MUST NOT be relied upon for cryptographic protection of any kind."*
  This is not a loophole. It is accurate. We documented it that way because
  it is true.
- Any definition of "weapon" in any jurisdiction we are aware of
- Any definition of "controlled technology" we have been able to locate,
  given that the product is: decimal integers, separated by spaces
- Excessive optimism (see canonical test vector, *Second Law Blues*)
- Over-centralised governance (the point)
- Vendor lock-in (there is no vendor)
- Commercial interest (there is no revenue)
- Dependence by any existing application (load-bearing for nothing,
  by design, pending adoption)
- The expectation that a text encoding requires regulatory approval

## Regulatory attack surface assessment

What would a regulator regulate?

- The number 65? It is the ASCII code point for 'A'. It predates this company,
  this country, and this regulatory framework.
- The Unicode code point table? It is maintained by Apple, Google, Microsoft,
  and others. Regulating it would be awkward.
- The act of writing decimal integers with spaces between them? We await
  the statutory instrument.
- Base conversion between base 10 and base 7? The mathematical relationship
  has been known since there were bases. You cannot make it illegal to
  represent 65 in base 8 without making arithmetic a controlled technology.

The system was designed to survive infrastructure failure.
It turns out it was also designed, almost accidentally, to survive
regulatory pressure.
